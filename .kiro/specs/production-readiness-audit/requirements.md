# Requirements Document

## Introduction

This document captures every issue found during a file-by-file production readiness audit of the QuiltCorgi codebase (Next.js 16 / React 19 / TypeScript / Tailwind v4 / Drizzle ORM / PostgreSQL / AWS Cognito). Every issue listed here was verified by reading the actual source files and re-verified for accuracy. Issues are grouped by category and ordered by severity within each group.

**Note:** This is the corrected version after verification. Removed false positives: S3 ContentLengthRange (already implemented), webhook Redis dedup (already implemented), community API session handling (misleading but functional), trust-engine tests (already exist).

---

## Glossary

- **System**: The QuiltCorgi web application and its API layer.
- **Studio**: The browser-based quilt design canvas (`/studio/[projectId]`).
- **Community**: The Social Threads feed and post detail pages.
- **Admin**: A user with `role = 'admin'` in the database.
- **Pro**: A user with `role = 'pro'` or `role = 'admin'` in the database.
- **Free**: A user with `role = 'free'` in the database.
- **Proxy**: `src/proxy.ts`, the Next.js middleware that handles JWT verification and route protection.
- **Webhook**: The Stripe webhook handler at `/api/stripe/webhook`.
- **Presigned URL**: A time-limited S3 upload URL generated by `/api/upload/presigned-url`.
- **CSP**: Content Security Policy header set in `next.config.ts`.

## Requirements

### Requirement 1: Infrastructure — Redis Rate Limiting

**User Story:** As an operator, I want distributed rate limiting across all server instances so that auth brute-force attacks are blocked even in a multi-instance deployment.

#### Acceptance Criteria

1. THE System SHALL provision Upstash Redis (`UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN`) before production launch.
2. THE System SHALL add both Redis env vars to the production AWS Secrets Manager secret.
3. WHEN `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are absent in production, `env-validation.ts` already emits a startup warning — the operator SHALL treat this as a launch blocker.
4. THE System SHALL NOT rely on the in-memory fallback rate limiter (`src/lib/rate-limit.ts`) in production, as it does not survive process restarts or span multiple instances.

---

### Requirement 2: Infrastructure — Stripe Live Keys

**User Story:** As an operator, I want Stripe configured with live keys before launch so that real payments can be processed.

#### Acceptance Criteria

1. THE System SHALL replace the `STRIPE_SECRET_KEY` placeholder in the production Secrets Manager secret with a live key (`sk_live_*`) before launch.
2. THE System SHALL register the webhook endpoint `https://quiltcorgi.com/api/stripe/webhook` in the Stripe Dashboard and subscribe to: `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`.
3. THE System SHALL update `STRIPE_WEBHOOK_SECRET` in the production secret with the live signing secret from the registered webhook.
4. THE System SHALL configure the Stripe Customer Portal with the production return URL (`https://quiltcorgi.com/profile/billing`).

---

### Requirement 3: Infrastructure — AWS Production Resources

**User Story:** As an operator, I want all AWS production resources provisioned and configured so that the app can store files, authenticate users, and serve content in production.

#### Acceptance Criteria

1. THE System SHALL provision a production Cognito User Pool (separate from dev) with `USER_PASSWORD_AUTH` flow, email verification, and a password policy, and write `COGNITO_USER_POOL_ID` and `COGNITO_CLIENT_ID` to Secrets Manager.
2. THE System SHALL create the production S3 bucket with private ACL, SSE-KMS encryption, and a CORS policy allowing PUT from the production domain only.
3. THE System SHALL set up a CloudFront distribution in front of S3 and set `NEXT_PUBLIC_CLOUDFRONT_URL` as a build-time environment variable in the deployment pipeline.
4. THE System SHALL set `NEXT_PUBLIC_APP_URL=https://quiltcorgi.com` as a build-time environment variable in the deployment pipeline.
5. THE System SHALL run `npm run db:migrate` against the production RDS instance before the first deploy.
6. THE System SHALL enable automated RDS backups with a minimum 7-day retention period.

2. THE System SHALL remove `src/app/api/reports/[id]/` — the directory exists with no `route.ts` file. The `reports` table was dropped in the same migration. The directory is dead code.

### Requirement 5: Dead Code — `StudioMobileGate` Wrapper

**User Story:** As a developer, I want no unnecessary wrapper components so that the component tree is easy to follow.

#### Acceptance Criteria

1. `src/components/mobile/StudioMobileGate.tsx` is a thin wrapper that renders `<StudioGate />` inside a `<div className="md:hidden">`. The studio page (`src/app/studio/[projectId]/page.tsx`) already wraps `<StudioClient>` in `<div className="hidden md:block">`, so the `md:hidden` wrapper in `StudioMobileGate` is redundant.
2. THE System SHALL inline `<StudioGate />` directly in the studio page (removing the `md:hidden` wrapper div since the `hidden md:block` on `StudioClient` already handles the split), and delete `StudioMobileGate.tsx`.

### Requirement 6: Incomplete Feature — Profile Stats Post Count Is Wrong

**User Story:** As a user, I want my profile stats to show my actual post count so that I can see how active I am in the community.

#### Acceptance Criteria

1. `src/app/profile/page.tsx` fetches `GET /api/community?limit=1` and reads `data.total` as the post count. This returns the total count of ALL approved community posts, not the current user's posts.
2. THE System SHALL fix the query to `GET /api/community?limit=1&creatorId=<userId>` so it counts only the current user's posts. The `creatorId` filter is already supported by the community API.

### Requirement 7: Incomplete Feature — Dashboard "Browse Fabrics" and "My Quiltbook" Cards Link to `/dashboard`

**User Story:** As a user, I want the dashboard bento cards to navigate to the correct destinations so that I can access my fabrics and projects directly.

#### Acceptance Criteria

1. In `src/app/dashboard/page.tsx`, the "Browse Fabrics" card (`<Link href="/dashboard">`) links to `/dashboard` (itself). It should navigate to the fabric library — either by opening the fabric panel in the Studio or navigating to a dedicated fabric browsing view.
2. In the same file, the "My Quiltbook" card also links to `/dashboard` (itself). It should scroll to or highlight the Recent Projects section, or link to a projects list view.
3. THE System SHALL fix both links to navigate to meaningful destinations before launch.

### Requirement 8: Incomplete Feature — System Fabric Images Are Placeholder SVGs

**User Story:** As a user, I want system fabrics to display real fabric swatches so that the fabric library is visually useful.

#### Acceptance Criteria

1. `src/db/seed/seedFabrics.ts` explicitly documents that `imageUrl` and `thumbnailUrl` for system fabrics are set to placeholder SVG data URIs (`data:image/svg+xml,...`) using approximate color hex values. The comment states: "In production, these would point to real images on S3/CloudFront."
2. THE System SHALL upload real fabric swatch images (or high-quality solid-color swatches) to S3/CloudFront for all system fabrics before launch.
3. THE System SHALL update the `imageUrl` and `thumbnailUrl` columns in the `fabrics` table for all system fabrics to point to the real URLs.

### Requirement 11: Security — CSP Missing Cognito JWKS Endpoint

**User Story:** As a security engineer, I want the Content Security Policy to allow the Cognito JWKS fetch so that JWT verification is not blocked by CSP.

#### Acceptance Criteria

1. `src/next.config.ts` sets `connect-src` to `'self' https://api.stripe.com https://accounts.google.com https://*.s3.amazonaws.com https://*.s3.*.amazonaws.com`. The Cognito JWKS endpoint (`https://cognito-idp.*.amazonaws.com`) is not included.
2. THE System SHALL add `https://cognito-idp.*.amazonaws.com` to the `connect-src` directive so JWKS fetches are not blocked by CSP in production.

---



### Requirement 12: Security — DB Connection Uses `rejectUnauthorized: false` in Production

**User Story:** As a security engineer, I want the database connection to verify the RDS TLS certificate so that the connection is not vulnerable to man-in-the-middle attacks.

#### Acceptance Criteria

1. `src/lib/db.ts` sets `ssl: { rejectUnauthorized: false }` in production. This disables TLS certificate verification.
2. THE System SHALL set `ssl: { rejectUnauthorized: true }` in production once the RDS certificate chain is trusted (AWS RDS uses a known CA bundle that can be bundled with the app).

---

### Requirement 13: Performance — Sitemap Has Duplicate `/blog` Entry

**User Story:** As an SEO engineer, I want the sitemap to be free of duplicate entries so that search engines do not receive duplicate content signals.

#### Acceptance Criteria

1. `src/app/sitemap.ts` contains two identical entries for `${BASE_URL}/blog` in the `staticPages` array (lines 27–34 and 35–39 are identical).
2. THE System SHALL remove the duplicate `/blog` entry.

---

### Requirement 14: Configuration — `robots.txt` Missing `/auth/` Disallow

**User Story:** As an SEO engineer, I want auth pages excluded from crawling so that sign-in and sign-up pages are not indexed.

#### Acceptance Criteria

1. `public/robots.txt` disallows `/api/`, `/admin/`, `/studio/`, and `/profile/` but does NOT disallow `/auth/`. Auth pages (`/auth/signin`, `/auth/signup`, etc.) should not be indexed.
2. THE System SHALL add `Disallow: /auth/` to `robots.txt`.

---

### Requirement 15: Configuration — PWA Manifest Uses Wrong Icon Files

**User Story:** As a mobile user, I want the PWA manifest to reference correctly sized icons so that the app installs cleanly on iOS and Android home screens.

#### Acceptance Criteria

1. `public/manifest.json` lists `logo.png` as both `192x192` and `512x512`. `logo.png` is not 192×192 or 512×512 — it is the brand logo at an unspecified size.
2. `public/icon-192.png` exists and is referenced in `layout.tsx` as the apple-touch-icon. A `icon-512.png` does NOT exist in `public/`.
3. THE System SHALL update `manifest.json` to reference `icon-192.png` for the 192×192 entry.
4. THE System SHALL create a proper `icon-512.png` (512×512 PNG) and add it to `public/` and `manifest.json`.

---



### Requirement 16: Observability — No Error Tracking

**User Story:** As an operator, I want structured error tracking in production so that I can detect, diagnose, and alert on application errors without manually tailing logs.

#### Acceptance Criteria

1. `src/instrumentation.ts` loads secrets and validates env vars at startup but does NOT initialize any error tracking SDK.
2. Several API routes use bare `console.error` with no aggregation: `src/app/api/community/route.ts`, `src/app/api/stripe/webhook/route.ts`, `src/app/api/stripe/checkout/route.ts`, `src/app/api/stripe/portal/route.ts`.
3. THE System SHALL integrate an error tracking service (e.g., Sentry) and instrument `src/instrumentation.ts` to initialize it at server startup.
4. THE System SHALL replace bare `console.error` calls in API routes with structured logging that includes the route and a sanitized error message.
5. THE System SHALL NOT log sensitive user data (email, tokens, payment info) in error messages.

---

### Requirement 17: Legal / Compliance — Account Deletion Not Implemented

**User Story:** As a user, I want to be able to delete my account and have my data removed so that I can exercise my right to erasure.

#### Acceptance Criteria

1. `src/app/api/profile/route.ts` implements `GET` and `PUT` but NOT `DELETE`. There is no account deletion endpoint.
2. The Privacy Policy (`src/app/(public)/privacy/page.tsx`) explicitly promises: "You can delete your projects and account. Upon account deletion, we will remove your personal data in accordance with our retention policy."
3. THE System SHALL implement a `DELETE /api/profile` endpoint that removes or anonymizes the user's personal data, cancels any active Stripe subscription, and deletes or anonymizes the user's S3 assets.
4. THE System SHALL retain community posts and comments in anonymized form (replace `userId` with null) if they have been interacted with by other users, per the Privacy Policy.
5. THE System SHALL expose an account deletion UI in the profile settings page.

---

### Requirement 18: Legal / Compliance — GDPR Data Export Not Implemented

**User Story:** As a user, I want to export all my personal data so that I can exercise my right to data portability.

#### Acceptance Criteria

1. The Privacy Policy promises: "You can export your quilt designs in multiple formats (PDF, PNG, SVG)." This refers to design export, not a personal data export. However, Section 6 also lists "Data portability" as a right users can exercise by contacting support.
2. THE System SHALL implement a data export endpoint (or a support-driven process) that returns a JSON archive of the user's account data, projects, community posts, and comments.
3. THE System SHALL NOT include other users' data in the export.

### Requirement 19: Copy / UX — Help Panel Pro Description Missing Photo to Pattern

**User Story:** As a user, I want the Help Panel to accurately describe what Pro includes so that I understand the value of upgrading.

It does NOT mention Photo to Pattern, which is described in `product.md` as the marquee Pro feature.

### Requirement 20: Testing — Missing Unit Tests for Critical Business Logic

**User Story:** As a developer, I want unit tests for the payment and session logic so that billing and auth bugs are caught before they affect users.

#### Acceptance Criteria

1. `tests/unit/lib/trust-engine.test.ts` exists and covers role/permission combinations — this is complete.
2. THE System SHALL have unit tests for the Stripe webhook handler covering: `checkout.session.completed` sets role to `pro`, `customer.subscription.deleted` sets role to `free`, `invoice.payment_failed` sets status to `past_due`. Currently only helper logic is tested, not the full endpoint integration.
3. THE System SHALL have unit tests for `cognito-session.ts` covering token refresh on expiry and graceful null return on refresh failure. No test file exists for this module.

---

### Requirement 21: Testing — No E2E Tests for Auth or Stripe Flows

**User Story:** As a developer, I want E2E tests for the critical auth and payment flows so that regressions are caught before deployment.

#### Acceptance Criteria

1. The Playwright config exists (`playwright.config.ts`) but there are no E2E test files in the project.
2. THE System SHALL have Playwright E2E tests covering: sign up → verify email → sign in → sign out.
3. THE System SHALL have Playwright E2E tests covering: unauthenticated user attempts protected route → redirected to sign-in → signs in → redirected back.
4. THE System SHALL run `npm run type-check` and `npm run lint` in CI with zero errors before every deploy.
